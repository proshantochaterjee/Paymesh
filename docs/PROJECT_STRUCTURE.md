## Module boundary rules

1. `apps/frontend` imports from `packages/shared` and `packages/sdk` only вЂ”
   never reaches into `apps/backend/src` or Prisma directly.
2. `apps/backend` imports from `packages/shared` and `packages/sdk` вЂ” never
   imports frontend code.
3. `packages/sdk` imports from `packages/shared` only вЂ” it must not depend
   on NestJS or Next.js so it stays usable from scripts and tests.
4. `packages/contracts` has no dependency on the TypeScript side; the
   relationship is one-way via generated bindings checked into
   `packages/sdk/src/contracts` (regenerated by
   `scripts/generate-contract-bindings.sh` whenever a contract's public
   interface changes).
5. Nothing outside `packages/contracts` and the Soroban network itself may
   move funds. This is enforced by code review, not tooling, and is the
   single most important boundary in the repository вЂ” see
   [SECURITY_MODEL.md](./SECURITY_MODEL.md).

## Naming conventions

- Rust contract crates: kebab-case directories, snake_case contract names
  (e.g., `packages/contracts/payroll-engine` compiles to `payroll_engine`).
- NestJS modules: one directory per bounded context under
  `apps/backend/src/modules/<name>`, each with `*.module.ts`,
  `*.controller.ts`, `*.service.ts`, `dto/`, and `*.spec.ts` files.
- Frontend feature folders mirror backend module names 1:1 (`employees`,
  `payroll`, etc.) so a contributor can find the frontend/backend/contract
  code for a feature by name alone.
